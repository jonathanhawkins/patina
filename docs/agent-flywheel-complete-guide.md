# The Complete Flywheel Guide - Agent-Based Software Development Methodology

## Overview

This comprehensive guide presents the "Agentic Coding Flywheel," a methodology for building software by orchestrating AI agent swarms using detailed markdown planning, task management systems called "beads," and coordinated execution frameworks.

## Core Methodology Phases

### Phase 1: Planning (85% of Effort)

The methodology invests heavily in planning before implementation begins. Rather than diving into code, developers:

1. **Create comprehensive markdown plans** describing the entire system's architecture, workflows, and user interactions
2. **Generate competing plans** using multiple frontier models (GPT Pro, Claude Opus, Gemini, Grok Heavy)
3. **Synthesize the best ideas** across competing proposals into a single superior hybrid plan
4. **Iterate ruthlessly** through 4-5+ refinement rounds until reaching 3,000-6,000+ lines of detailed specification

The rationale: "A markdown plan, even a massive 6,000-line one, is still vastly smaller than the codebase it describes." This enables holistic system reasoning while changes remain cheap.

### Phase 2: Converting Plans to Beads

"Beads" are self-contained work units similar to Jira tasks but optimized for AI agent execution. Each bead includes:

- Detailed descriptions with embedded reasoning
- Complete testing specifications and acceptance criteria
- Explicit dependencies on other beads
- Context sufficient for agents to execute without referencing the original plan

A typical complex project generates 200-500 beads with full dependency graphs.

### Phase 3: Bead Polishing

Before implementation, beads undergo 4-6+ refinement rounds using the principle: "Check your beads N times, implement once." Each pass identifies duplicates, missing dependencies, incomplete context, and ensures comprehensive coverage of the original plan.

### Phase 4: Agent Swarm Execution

**Coordination Infrastructure:**

- **Beads (br tool)**: Durable task definitions with dependency graphs
- **Agent Mail**: Targeted messaging between agents (not broadcast)
- **bv**: Graph-theory routing that computes optimal task priority using PageRank and betweenness analysis

"When multiple agents each independently query bv for priority, you get emergent coordination."

### Phase 5: Review, Testing & Hardening

Implementation includes:

- **Fresh eyes self-review** after each bead completion
- **Cross-agent review** to catch integration issues
- **Comprehensive testing**: Unit tests and e2e tests written alongside implementation
- **UI/UX polish** as a dedicated phase after core functionality works

## Critical Tools & Patterns

### AGENTS.md: The Operating Manual

Every agent reads a project-specific AGENTS.md file containing:

- Safety constraints (no destructive git commands, no file deletion without permission)
- Tool documentation
- Project-specific rules and conventions
- Behavioral expectations

"After every context compaction, agents must re-read it." This is the single most common intervention: "Reread AGENTS.md so it's still fresh in your mind."

### Single-Branch Git Model

All agents work directly on `main` rather than using feature branches or worktrees. Coordination mechanisms prevent conflicts:

- **File reservations** via Agent Mail (advisory, not rigid, with TTL expiry)
- **Pre-commit guards** that block commits to reserved files
- **DCG (Destructive Command Guard)**: Mechanical blocking of dangerous commands like `git reset --hard`

### Fungible Agents

No specialized roles. All agents are generalists reading the same AGENTS.md and able to pick up any bead. This prevents single points of failure: when one agent crashes or needs compaction, others simply resume its work.

## Real-World Example: CASS Memory System

The methodology proved itself with the CASS Memory System:

- **5,500-line markdown plan** synthesized from 4 frontier models
- **347 beads** with full dependency structure
- **11,000 lines of working code** generated by 25 agents
- **204 commits** in approximately 5 hours
- Production-ready, tested implementation

## Three Reasoning Spaces

The methodology separates work into distinct layers, each with different cost characteristics:

| Space | Artifact | Cost of Fixes | Purpose |
|-------|----------|---------------|---------|
| **Plan** | Markdown document | 1x | Architecture, workflows, system-wide decisions |
| **Bead** | Task definitions | 5x | Execution boundaries, dependencies, context |
| **Code** | Source files | 25x | Implementation and verification |

The key insight: catching errors in plan space costs far less to fix than discovering them during implementation.

## Major Features: Research & Reimagine Process

For adding substantial capabilities to existing projects, the methodology includes a specialized seven-step process:

1. Investigate and propose by studying relevant external projects
2. Iteratively deepen the proposal beyond conservative initial suggestions
3. Perform inversion analysis to identify uniquely possible capabilities
4. Run 5x repeated "blunder hunts" to catch logical flaws
5. Close specific design gaps with concrete decisions
6. Prepare self-contained background documentation
7. Obtain multi-model feedback and synthesize results

This approach was successfully applied to integrate NATS messaging architecture into the Asupersync project.

## Practical Operational Setup

### Agent Composition

For different project phases:

- **Planning**: GPT Pro (Extended Reasoning)
- **Plan synthesis**: GPT Pro
- **Bead creation**: Claude Code (Opus)
- **Implementation**: Claude Code + Codex + Gemini (ratio typically 2:1:1)
- **Review**: Claude Code + Gemini

### Swarm Launch

Use `ntm` (Named Tmux Manager) or WezTerm to manage multiple agent sessions:

```bash
ntm spawn myproject --cc=2 --cod=1 --gmi=1
ntm send myproject "Marching orders prompt"
```

Agents should be staggered in launch (30+ seconds apart) to avoid "thundering herd" contention on initial task selection.

### Monitoring & Maintenance

Human operators spend ~10-30 minutes per cycle:

- Checking bead progress with `br list` and `bv --robot-triage`
- Managing compactions by sending "reread AGENTS.md" prompts
- Triggering periodic reviews
- Handling rate limiting with account switching
- Creating new beads for unanticipated issues

## Key Insights

**Planning compounds returns**: Each planning cycle produces better artifacts (plans, then beads) that feed into faster, safer implementation.

**Specificity in artifacts, generality in prompts**: Reusable prompts work across all projects because project details live in beads and AGENTS.md, not in prompts themselves.

**Security emerges from completeness**: Thorough architectural planning naturally surfaces security gaps, authentication issues, and data exposure risks without requiring a separate security checklist.

**Strategic drift detection**: The methodology guards against busy swarms heading in wrong directions. Regular reality checks ensure bead graphs actually converge on project goals: "Where are we on this project? Do we actually have the thing we are trying to build?"

## When Planning Breaks Down

If heavy cognitive work becomes necessary during implementation, it signals planning or bead insufficiency. The specific remedies:

- **Vague beads** → agents improvise inconsistently
- **Missing dependencies** → agents work on blocked tasks
- **Thin AGENTS.md** → agents produce non-idiomatic code
- **No Agent Mail** → agents create file conflicts

The solution always involves returning to bead space and adding missing detail rather than pushing through implementation.
